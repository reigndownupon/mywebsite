Project / Learning Program

Assembly by Hand

A 30 day, hands-on x86-64 assembly path for learning computer science fundamentals from the metal up. Each day now includes the practical reason for the lesson and what to notice as each command runs.

Start Day 1 Browse all 30 days

How it works

Not a quiz app. A lab notebook.

The lessons are self-guided. You read the day, type the assembly by hand, run the commands on the VPS, inspect what changed, and keep notes. The goal is to connect every command to a concrete thing: an object file, an executable, a register, a memory address, output, or a test result.

30 day index

From syscalls to a hash map.

Days 1 to 19 build the machine model. Days 20 to 30 build, test, explain, and extend the hash map. Open any day to see the outcome, practical application, command lessons, lab, and completion target.

Day 01 CPU, memory, registers, and your first syscall

Outcome: Understand what assembly is actually controlling: registers, memory, instructions, syscalls.

Practical application: You learn that programs are not magic. They become instructions that place values in registers and ask the operating system to do work.

What to learn from the commands: make clean removes old build artifacts so you know the result came from today’s source. make day-01 assembles, links, runs the program, and checks that the OS received exit code 42. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Write a Linux x86-64 program that exits with status 42 using only _start, mov, and syscall.

Done when: You can assemble, link, run, and inspect $?.

Day 02 The build pipeline: source to object to executable

Outcome: Learn what assembler, linker, ELF, symbols, and sections do.

Practical application: You learn how source code becomes an executable file, which is the foundation for debugging compilers, linkers, malware, packed binaries, and broken builds.

What to learn from the commands: gcc -nostdlib teaches what happens when you remove the normal C runtime. objdump -d shows the actual machine instructions. readelf -h shows the executable format the OS loader reads. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Build the same program with gcc -nostdlib, inspect it with objdump -d and readelf -h.

Done when: You can explain source, object, executable, symbol, section.

Day 03 Registers and data movement

Outcome: Learn general purpose registers, register sizes, zero extension, and mov rules.

Practical application: You learn to treat registers like the CPU’s working desk. Most high-level variables become register moves and arithmetic at some point.

What to learn from the commands: The build command proves your register arithmetic survives assembly and linking. The exit status check teaches that rax and rdi values eventually become observable program behavior. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Write arithmetic in registers and return the result as process exit code.

Done when: You can predict register values before running.

Day 04 Arithmetic and flags

Outcome: Use add, sub, imul, inc, dec, cmp, flags, and conditional jumps.

Practical application: You learn how decisions really happen. Every if statement eventually becomes comparison, flags, and jumps.

What to learn from the commands: The run command teaches whether your branch went where you expected. Disassembly teaches how labels and conditional jumps replace high-level control flow. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Write a program that computes a small expression and branches on the result.

Done when: You can describe how comparison is subtraction plus flags.

Day 05 Memory addressing by hand

Outcome: Learn bytes, words, doublewords, quadwords, labels, RIP relative addressing, and effective addresses.

Practical application: You learn that memory is addressed byte by byte. Strings, arrays, structs, files, packets, and heap objects all depend on this model.

What to learn from the commands: make day-05 builds and runs a raw write syscall. Seeing A printed teaches that rsi held an address and rdx held a byte count. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Store values in .data, load them, modify them, and write a byte to stdout.

Done when: You can read and write memory without treating it like magic.

Day 06 The stack

Outcome: Learn rsp, push, pop, stack growth, alignment, and temporary storage.

Practical application: You learn the stack as temporary storage and call history. This is essential for debugging crashes, understanding exploits, and reading function calls.

What to learn from the commands: The day target should prove your pushes and pops restore values in the right order. Single-stepping teaches that the stack grows downward as rsp changes. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Reverse three values using only the stack.

Done when: You can draw the stack before and after every instruction.

Day 07 Control flow patterns

Outcome: Translate if, while, for, and do while into labels and jumps.

Practical application: You learn how loops are built from labels, counters, comparisons, and jumps. This is the skeleton beneath for, while, retries, parsers, and scanners.

What to learn from the commands: Running the target checks the loop result. Disassembly teaches that there is no special loop object, only state plus a jump back. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Implement a loop that sums 1 through N.

Done when: You can translate simple C control flow into assembly.

Day 08 Functions and the System V ABI

Outcome: Learn calls, returns, argument registers, return values, caller saved vs callee saved.

Practical application: You learn how assembly cooperates with C. This is the bridge to real programs, libraries, operating system APIs, and mixed-language debugging.

What to learn from the commands: Compiling the C harness teaches where arguments are placed before the call. The test proves your assembly returned the correct value in rax. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Write add3(a,b,c) in assembly and call it from C.

Done when: You can call assembly from C using the ABI.

Day 09 Stack frames and local variables

Outcome: Learn rbp, prologues, epilogues, local stack slots, and alignment.

Practical application: You learn how local variables and call frames are organized. This makes stack traces, crash dumps, and debugger views understandable.

What to learn from the commands: The build and debugger commands teach how rbp can anchor a stable view of locals even while rsp moves. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Write a function with two local variables and inspect stack layout.

Done when: You can debug stack variables with gdb or objdump.

Day 10 Strings and bytes

Outcome: Learn null termination, byte loops, lea, and pointer incrementing.

Practical application: You learn pointer walking with a real primitive. Many parsers, protocols, and C library routines are just careful byte loops.

What to learn from the commands: make day-10 compiles assembly plus a C test harness. The tests teach that an empty string, one-character string, and normal string all end at the null byte. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Implement strlen by hand.

Done when: Your assembly strlen passes a C test harness.

Day 11 Arrays and pointer arithmetic

Outcome: Learn scaled indexing, element sizes, and base plus index addressing.

Practical application: You learn that arrays are base address plus element size times index. This is the heart of buffers, tables, packet fields, and memory safety.

What to learn from the commands: The target should prove your scaled addressing matches the C array layout. Each test teaches that count bounds matter as much as the pointer. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Implement sum_i64(int64_t *xs, size_t n).

Done when: You can walk arrays safely with pointer and count.

Day 12 Structs and layout

Outcome: Learn padding, alignment, field offsets, and how C structs become bytes.

Practical application: You learn that structs are agreed-upon byte offsets. This is crucial for ABIs, file formats, network structures, and reverse engineering.

What to learn from the commands: The C harness teaches the real offsets. Your assembly load instructions teach how field names become numbers like 0, 8, and 16. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Create a C struct and write an assembly function that reads its fields.

Done when: You can calculate offsets by hand and verify with offsetof.

Day 13 Debugging assembly

Outcome: Use gdb, breakpoints, registers, memory examine, and single stepping.

Practical application: You learn to debug instead of guess. Assembly rewards patient observation of registers, memory, and flags.

What to learn from the commands: gdb commands teach one instruction at a time. Register and memory inspection teaches cause and effect directly. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Debug a deliberately broken function and write notes on the root cause.

Done when: You can use tools instead of guessing.

Day 14 Bit operations

Outcome: Learn and, or, xor, not, shifts, rotates, masks, and alignment tricks.

Practical application: You learn bit-level control. Masks and shifts drive permissions, flags, alignment, compression, crypto plumbing, and protocol parsing.

What to learn from the commands: The tests teach whether your mask keeps or removes the intended bits. Edge cases teach why zero and powers of two need special attention. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Implement power of two check and bit count starter.

Done when: You can explain how flags and masks build low level logic.

Day 15 Unsigned and signed numbers

Outcome: Learn two’s complement, sign extension, signed vs unsigned jumps, overflow.

Practical application: You learn that the same bits can mean different numbers. Signedness bugs are common in security, parsing, and boundary checks.

What to learn from the commands: Separate signed and unsigned tests teach when to use jl versus jb. Edge values teach why overflow changes interpretation. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Write min/max for signed and unsigned inputs and test edge cases.

Done when: You know when to use jl vs jb.

Day 16 Calling libc safely

Outcome: Learn external symbols, printf, stack alignment before calls, and varargs basics.

Practical application: You learn how to call real library functions safely. This is where hand-written assembly starts joining normal software ecosystems.

What to learn from the commands: The link command teaches external symbols. Running printf teaches stack alignment and why xor eax, eax matters for variadic calls. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Call printf from assembly with integer output.

Done when: You can use libc without corrupting the stack.

Day 17 Dynamic memory from C

Outcome: Use malloc, free, and understand heap pointers from assembly.

Practical application: You learn heap ownership from assembly. Allocation, use, and freeing become explicit instead of hidden behind high-level syntax.

What to learn from the commands: malloc returning a pointer teaches how heap memory enters your program. free teaches that ownership must be completed, not just obtained. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Allocate an array, fill it, print values, and free it.

Done when: You can cross from assembly into runtime services.

Day 18 Building a tiny standard library

Outcome: Implement reusable strlen, strcmp, memcpy, and memset subsets.

Practical application: You learn to build reusable primitives. Serious systems are built by composing small, tested routines.

What to learn from the commands: Creating a tiny library teaches object files and archives. Running tests against it teaches that reusable code needs stable behavior, not just one demo. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Create libtinyasm.a and link tests against it.

Done when: You start thinking in reusable primitives.

Day 19 Hashing fundamentals

Outcome: Understand deterministic hashing, byte mixing, modulo/capacity, and collisions.

Practical application: You learn how strings become deterministic numbers. Hashing is the doorway into hash maps, caches, sets, deduplication, and indexing.

What to learn from the commands: make day-19 compares your FNV-1a output to known vectors. Matching vectors teaches that every byte and multiply must be exactly right. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Implement FNV-1a hash for null terminated strings.

Done when: Your hash output matches known test vectors.

Day 20 Hash table design

Outcome: Learn buckets, entries, keys, values, capacity, load factor, and separate chaining.

Practical application: You learn to design memory before writing code. A hash map is mostly a set of layout and ownership decisions made visible.

What to learn from the commands: Reading the constants teaches how C fields become assembly offsets. make day-20 runs the current hash map tests so the design stays executable. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Design the structs and constants for a string to int hash map.

Done when: You can draw memory layout for the table.

Day 21 Hash map initialization

Outcome: Write hm_init to allocate buckets and initialize fields.

Practical application: You learn initialization and cleanup, two habits that separate toy code from reliable systems code.

What to learn from the commands: make day-21 exercises hm_init and hm_free. The tests teach that valid capacity, zeroed buckets, size zero, and cleanup are observable invariants. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Implement init/free skeleton with C harness.

Done when: A new map has valid capacity, size 0, and null buckets.

Day 22 Entry allocation and ownership

Outcome: Handle key copying, value storage, next pointers, and failure paths.

Practical application: You learn safe allocation paths. Real low-level code must handle the moment where one allocation succeeds and the next one fails.

What to learn from the commands: The hash map test teaches that entries own copied keys. ASAN teaches whether cleanup avoids leaks and invalid frees under normal paths. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Implement entry creation with malloc and strdup or manual copy.

Done when: No leaks on allocation failure.

Day 23 Hash map insert

Outcome: Implement bucket index, collision chain traversal, update existing key, insert new key.

Practical application: You learn insertion through hashing, indexing, and pointer linking. This is where abstract data structures become memory operations.

What to learn from the commands: The target teaches that hash modulo capacity selects a bucket. Collision tests teach that prepending a node is just pointer rewiring. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Write hm_put(map,key,value).

Done when: Inserts and updates pass tests.

Day 24 Hash map lookup

Outcome: Implement search through a bucket chain and return found status/value.

Practical application: You learn lookup and update. Efficient data structures are about finding the right memory without scanning everything.

What to learn from the commands: The tests teach hit versus miss behavior. The update case teaches that changing an existing value must not grow map.size. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Write hm_get(map,key,out_value).

Done when: Lookups pass hit, miss, and collision tests.

Day 25 Hash map delete

Outcome: Unlink nodes in singly linked chains, free key and entry, update size.

Practical application: You learn deletion in linked structures. Removing a node safely is one of the clearest tests of pointer discipline.

What to learn from the commands: The delete tests teach head, middle, tail, and missing-key cases. Each case shows why you track the link that points to the node, not only the node. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Write hm_delete(map,key).

Done when: Delete head, middle, tail, and missing keys correctly.

Day 26 Resizing and rehashing

Outcome: Grow capacity when load factor is high; move nodes without reallocating keys.

Practical application: You learn integration. A data structure is not finished when one function works. It is finished when the operations work together without corrupting ownership.

What to learn from the commands: make test-hashmap checks behavior. make test-hashmap-asan checks memory safety signals such as invalid access and leaks. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Implement hm_resize and integrate into put.

Done when: Map remains correct after many inserts.

Day 27 Testing assembly like production code

Outcome: Use C test harnesses, sanitizers where possible, randomized tests, and invariants.

Practical application: You learn that tests are executable understanding. They document what your assembly promises to do.

What to learn from the commands: Adding a test before code teaches discipline. Running the suite after every change teaches whether your mental model matches the machine. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Day 28 Performance and cache intuition

Outcome: Learn pointer chasing, branch prediction, locality, capacity choice, and profiling basics.

Practical application: You learn performance only after correctness. Speed work is meaningful when you can prove the code still behaves.

What to learn from the commands: The benchmark commands teach timing, but the first command is still correctness. Passing tests first prevents performance work from hiding memory bugs. Practical application: You learn packaging and handoff. A fresh clone should be able to build, test, and understand the project without you explaining it live.

Day 30 Teach it back and extend it

Outcome: Solidify mastery by explaining and extending your map.

Practical application: You learn mastery by teaching and extending. If you can explain every instruction and safely add a feature, the knowledge is becoming yours.

What to learn from the commands: The final commands should prove the extension did not break the baseline. The written explanation teaches whether you understand the why, not only the what. As you run each command, pause and name what changed: source file, object file, executable, register value, memory value, exit code, printed output, or test result.

Lab: Write a guided explanation of your hash map and add one feature: iteration, string values, or custom allocator.

Done when: You can explain every instruction and every byte of layout.
